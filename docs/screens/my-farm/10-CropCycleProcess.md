# The Complete Crop Cycle Process (Stage-by-Stage)

> **Tab:** My Farm · **Stack:** `MyFarmStack` (re-registered in `AIStack` for deep-linking) · **Route name:** *(none — this is a cross-screen reference, not a single screen)* · **File:** [`frontend/src/screens/FarmProfile/`](../../../frontend/src/screens/FarmProfile/)

## Purpose
This is the **standalone, end-to-end reference** for the full real-world Indian crop cycle that My Farm now supports in real time — stage by stage, from before the seed is in the ground to the day the harvest is sold and the season is closed out. Every other doc in this folder (01–09) covers **one screen**; this doc covers **the whole process** and maps each real-world farming step to the exact **capture screen / field** and the **`farmApi` call** that records it.

It exists so a field worker, agronomist or developer can answer one question end-to-end: *"A farmer is at point X in their season — where in the app do they record it, and where does that data land?"* It is the "complete process a farmer can use in real time", written India-first (Kharif/Rabi/Zaid seasons, acre/guntha/bigha units, PMFBY claims, mandi/APMC sales, KCC credit, udhaar).

A **crop cycle** is the unit of work — **one crop, one plot, one season** — and it moves through **8 growth stages in fixed order**. The canonical order lives in [`StageTimelineBar.js`](../../../frontend/src/screens/FarmProfile/ui/StageTimelineBar.js) (`STAGES` array) and matches the `GrowthStage` enum in [`schema.prisma`](../../../backend/prisma/schema.prisma):

```
PLANNING → LAND_PREP → SOWING → VEGETATIVE → FLOWERING → FRUITING → MATURITY → HARVESTED
  (Plan)    (Prep)     (Sow)     (Grow)       (Flower)    (Fruit)    (Mature)   (Harvest)
```

> **`FRUITING` is horticulture language; read it per crop category.** The enum value is fixed, but the *displayed* label should map to the crop's botany: **"Grain filling"** for cereals (wheat, rice, maize, jowar, bajra), **"Pod development"** for pulses/oilseeds (tur, gram, soybean, groundnut, sunflower), **"Boll development"** for cotton, and **"Fruit set / development"** for vegetables and fruit crops. The generic "Fruit" caption in `StageTimelineBar` and `GrowthStoryScreen` is fine as a fallback, but any per-crop copy should use the category-correct term. *(Label-mapping is a Notes/Future-work item — the enum and timeline labels are crop-agnostic today.)*

Two interaction shapes run the cycle, and this doc keeps them separate throughout:

1. **The Crop Plan** — a one-time, guided, **pre-seeding** capture ([`CropCycleCreateScreen.js`](../../../frontend/src/screens/FarmProfile/CropCycleCreateScreen.js), doc [05](05-CropCyclePlanPage.md)). It is a **single ScrollView** of eight numbered sections (not a multi-step wizard — the progress bar is just `requiredDone/3` for crop + season + area). It captures the field's **previous crop / rotation**, **field-preparation** plan, **water source**, and the **seed block incl. seed treatment + product**, then creates the cycle in the `PLANNING` stage. It does **not** itself set `sowingDate` — DAS starts later, when the farmer logs sowing.
2. **The ongoing log** — repeatable, dated, event-based entries appended to JSON arrays on the cycle (each irrigation, each fertiliser dose, each spray) for the rest of the season, captured from the activity picker ([07](07-ActivityTypePickerPage.md)) and the loggers/modals ([06](06-CropCycleDetailPage.md), [08](08-ActivityLoggers.md)).

- **Reached from:** this is a *process map*, not a navigable screen. The process is walked by the farmer across the My Farm screen flow:
  - **Stage 0 (plot header)** → [`FarmAddEditScreen.js`](../../../frontend/src/screens/FarmProfile/FarmAddEditScreen.js) (doc [04](04-FarmAddEditPage.md)).

- **Route params in:** —. Each underlying screen threads `farmId` (the plot) and then `cycleId` (the cycle) as the farmer moves through the process.

## How it works
**One spine, two clocks, one truth.**

- **The stage spine** is the 8-stage `GrowthStage` enum. The **live, authoritative stage is always `cycle.growthStage`**, set manually by the farmer via `advanceStage(cycleId, stage)`. It only moves forward when the farmer says the crop reached that phase.
- **The DAS clock** — **Days After Sowing** = `sowingDate ? max(0, floor((now − sowingDate) / 86400000)) : null` — starts the moment `sowingDate` is stamped. That stamp is set by **[`SowingLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/SowingLogScreen.js)**, not by the Crop Plan: when sowing is logged it calls `advanceStage('SOWING')` and then `updateCropCycle(cycleId, { sowingDate: now })` (`farmApi.updateCropCycle` → `PATCH /cycles/:id`). So a cycle created by the Crop Plan starts at `PLANNING` with **DAS = null** until the farmer logs sowing. DAS is the timeline spine for the dashboard hero, the cycle detail screen and the Growth Story. Source: [`CropCycleDetailScreen.js`](../../../frontend/src/screens/FarmProfile/CropCycleDetailScreen.js) (~line 297).
- **Derived stage is display-only.** DAS→stage bands are used *only* to bucket historical events into Growth Story chapters and to pick its hero scene; they are **approximate** and never override `cycle.growthStage`. They are also **crop- and sowing-date-sensitive** — a late-sown rabi crop compresses the whole calendar — so prefer a per-crop DAS table over any single hard-coded prose example. As a worked illustration, **generic wheat**: tillering / `VEGETATIVE` ~0–60 DAS → jointing–booting ~60–80 → anthesis / `FLOWERING` ~80–100 → grain-fill (the cereal reading of `FRUITING`) ~100–125 → `MATURITY` ~125–140. *(The shipped `GrowthStoryScreen` anchors are mid-duration generic, not wheat-specific.)*

**The capture model.** The Crop Plan captures the field's **history and intent** — previous crop, the field-preparation plan, water source, and the seed block (incl. seed treatment + product) — and composes the prev-crop / field-prep / water inputs into a single `notes` line ("Previous crop: … · Field prep: … · Water: …") persisted on the cycle. The *actual* dated land-prep and sowing **events** are then logged from [`LandPrepLogScreen`](../../../frontend/src/screens/FarmProfile/logging/LandPrepLogScreen.js) / [`SowingLogScreen`](../../../frontend/src/screens/FarmProfile/logging/SowingLogScreen.js) once the cycle exists (sowing also stamps `sowingDate`). After that, every later operation — and *additional* prep passes or re-sowing — is an **add-multiple dated log** appended to a JSON array on the cycle (`irrigationLogs[]`, `fertilizersUsed[]`, `pesticidesUsed[]`, `observedEvents[]`, `activities[]`, `laborLogs[]`, `expenseLogs[]`, `incomeLogs[]`). The same operation recurs many times across the season; nothing after sowing is a single locked field.

**Cross-cutting design goals** (target shape for every operation, not all wired yet): each entry is **timestamped** and, once `sowingDate` exists, **DAS-stampable**; values **prefill from crop presets but every value is farmer-editable**; capture should be **offline-first** (log now, sync later). The richer affordances — per-cost **bill/receipt photo** (for PMFBY claims, dealer complaints, traceability), **GPS**, **voice-note (local language)** and an **"on credit (udhaar)?" toggle** — are the India-first direction this doc describes; treat them as **Notes/Future work** except where a specific logger already implements them. *(Notably the Crop Plan's seed block has no `seedReceiptUrl` photo capture yet — see Gaps.)*

### Stage-by-stage map — real-world step → capture screen/field → `farmApi` call

#### Stage 0 — Plot / cycle header *(pre-cycle)*
| Real-world step | Where it's captured | `farmApi` call → data |
|---|---|---|
| Register the plot (name, GPS boundary, total area + unit, land tenure, soil type, irrigated/rainfed) | [`FarmAddEditScreen.js`](../../../frontend/src/screens/FarmProfile/FarmAddEditScreen.js) (doc [04](04-FarmAddEditPage.md)); the cycle's season/year/area come from Crop Plan §1–3 | `createFarm` / `updateFarm`; then `season`, `year`, `areaAllocatedAcres` written by `createCropCycle` |

#### Stage 1 — PLANNING *(the Crop Plan — single ScrollView, sections 1–8)*
The shipped Crop Plan creates the cycle in `PLANNING`. **Required** = crop + season + area (the `requiredDone/3` progress bar); the rest are optional chips/fields. It does **not** advance the stage or set `sowingDate`.

| Real-world step | Where it's captured | `farmApi` call → data |
|---|---|---|
| **What are you growing?** crop (illustrated picker, fuzzy search — bhendi/kapas/soya), season tile (Kharif/Rabi/Zaid), area (acres) | Crop Plan **§1–3** | `createCropCycle` → `cropName`, `season`, `year` (current year), `areaAllocatedAcres` |
| **Variety & type** variety name, Hybrid/Desi, Organic | Crop Plan **§4** | `createCropCycle` → `variety` (also sent as `seedName`), `isHybrid` (= `seedType==='hybrid'`), `isOrganic` |
| **How was this field cultivated before?** previous crop / rotation (chips + free text, incl. "Fallow / first time") | Crop Plan **§5** | composed into `notes` → `createCropCycle` ("Previous crop: …") |
| **Field preparation** plan (ploughing, harrowing, levelling, FYM/compost, bunds/ridges, summer plough — multi-select) | Crop Plan **§6** | composed into `notes` ("Field prep: …") |
| **Water source** (canal, borewell, open well, rainfed, drip, farm pond) | Crop Plan **§7** | composed into `notes` ("Water: …") |
| **Seed details** brand, source (dealer/own/coop/govt), seed rate (kg), seed cost (₹), seed treatment (Treated / Not) + product chip (Trichoderma / Carbendazim / Imidacloprid / **Rhizobium / PSB** / Thiram, or free text) | Crop Plan **§8** | `createCropCycle` → `seedBrand`, `seedSource`, `seedQuantityKg`, `seedTotalCostInr`, `seedTreatment`, `seedTreatmentProduct` (product only sent when Treated) |

> **Seed-treatment order (agronomy).** When more than one dressing is used the field sequence is **fungicide → insecticide → bio-agent / biofertilizer LAST**, applied only **after the chemical dressings have dried** — a contact fungicide will kill a live inoculant if mixed wet. For **pulses, oilseeds and groundnut**, **Rhizobium + PSB** inoculation is a near-universal step (the "Rhizobium / PSB" chip), not just a generic "bio". *(Today the screen captures a single treatment flag + one product chip; the ordered fungicide→insecticide→bio sequence and per-product dose are not yet structured — Notes/Future work.)*

> **Not captured by the Crop Plan (schema-only gaps).** The `FarmCropCycle` row has columns the create service will persist *if sent*, but no screen writes them: `cropCategory`, `cropNameMr` / `cropNameHi` (trilingual crop names), `seedCostPerKgInr`, `seedPurchaseDate`, `seedReceiptUrl` (bill photo), `expectedHarvestDate`. Soil-test status (SHC) and a structured resource/credit/PMFBY plan are **not modelled at all** — describe them only as future work, not current behaviour. The backend `createCropCycle` *does* now persist `seedTreatment`, `seedTreatmentProduct`, `seedPurchaseDate` and `notes` (columns already existed; no migration).

#### Stage 2 — LAND_PREP *(every dated prep pass via `LandPrepLog`)*
The Crop Plan records the field-prep **plan** only (a `notes` line, §6 above). The dated land-prep **events** — including the very first one — are logged from the Land Prep logger; nothing in the Crop Plan writes a `LAND_PREP` activity.

| Real-world step | Where it's captured | `farmApi` call → data |
|---|---|---|
| **Pre-sowing irrigation — palewa / rauni** (the standard land-prep watering that brings the seed-bed to working moisture before tillage/sowing) | [`IrrigationLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/IrrigationLogScreen.js) or [`LandPrepLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/LandPrepLogScreen.js) | `addIrrigationLog` / `addActivity(cycleId,{type:'LAND_PREP',…})` |
| **Field / seed-bed prep passes** (summer/deep plough, harrow, rotavator, levelling, bund, ridge-furrow, puddling, stubble removal) — implement/power, passes, hired/own, machine + diesel cost, labour | [`LandPrepLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/LandPrepLogScreen.js) | `addActivity(cycleId,{type:'LAND_PREP',…})` + `addLaborLog` (labour) + `addExpenseLog` (machinery) |
| Amendment incorporation (FYM/vermicompost/poultry/green-manure/pressmud/neem-cake/gypsum/lime/silt — qty, source, material + transport cost) | `LandPrepLog` | `addActivity` + `addExpenseLog` |

> The Crop Plan's §6 field-prep chips are the **plan**; each `LandPrepLog` entry is an **actual dated pass**. Re-prep, second passes and amendment incorporation are all repeatable logs.

#### Stage 3 — SOWING *(every sowing event via `SowingLog`; the seed *block* came from the Crop Plan)*
The **seed details** (brand, source, rate, cost, treatment + product) are captured once in the Crop Plan §8 and persisted on the cycle. The **sowing event itself** is logged from the Sowing logger, which is what advances the stage and starts the clock.

| Real-world step | Where it's captured | `farmApi` call → data |
|---|---|---|
| **Seed selection & source** (brand, source, seed rate kg, cost, treatment) | Crop Plan **§8** (see Stage 1) | `createCropCycle` → `seedBrand`, `seedSource`, `seedQuantityKg`, `seedTotalCostInr`, `seedTreatment`, `seedTreatmentProduct` |
| **Sowing / transplant event** (method broadcast/line/dibbling/transplant, seed used kg, sowing labour cost, notes) | [`SowingLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/SowingLogScreen.js) | `addActivity(cycleId,{type:'SOWING',fields:{method,seedKg}})` → then `advanceStage('SOWING')` → then `updateCropCycle(cycleId,{sowingDate:now})` (**starts the DAS clock**); labour → `addLaborLog` |
| **Re-sowing / gap-filling** (later passes) | [`SowingLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/SowingLogScreen.js) via the picker | `addActivity(cycleId,{type:'SOWING',…})` |

> Logging sowing is the pivot: it enters the `SOWING` stage, stamps `sowingDate` (**starts DAS**), and "begins the film" for Growth Story. Row×plant spacing, depth, soil-moisture (Dry/Optimum *vapsa*/Wet), DSR/SRI method, a nursery step (age at transplant, seedlings raised) and `expectedHarvestDate` are **not captured today** — Notes/Future work.

#### Stages 4–7 — VEGETATIVE · FLOWERING · FRUITING · MATURITY *(continuous ops, logged at any DAS)*
These stages are **not stage-locked screens** — every operation is logged at any DAS, and the *stage* is set by the farmer (`advanceStage`) for display. Two entry points: the **quick-log rail** on [`CropCycleDetailScreen.js`](../../../frontend/src/screens/FarmProfile/CropCycleDetailScreen.js) opens **five inline modals** (Water, Fertilize, Spray, Harvest, Sale) and pushes the rest; the **[activity picker grid](07-ActivityTypePickerPage.md)** instead pushes a `logging/*` screen for most types and routes only `FERTILIZER`/`SPRAY`/`HARVEST`/`SALE` to `CropCycleDetail` (via `prefillActivity`) — so irrigation is a *modal* from the rail but a *pushed logger* from the grid. *(`FRUITING` here reads as Grain-filling / Pod-development / Boll-development / Fruit-set per crop category — see the note above.)*

| Real-world step | Where it's captured | `farmApi` call → data |
|---|---|---|
| **Fertiliser / nutrient dose** (Basal vs Top-dress 1–2 / Foliar / Fertigation; Urea/DAP/MOP/SSP/NPK/ZnSO₄/Borax/organic; qty, method, bill) | **inline `FERTILIZER` modal** on `CropCycleDetail` (`setModal('fertilizer')`) | `addFertilizer(cycleId, entry)` → `fertilizersUsed[]` |
| **Irrigation event** (method flood/furrow/drip/sprinkler/rain-gun/rainfed; criticality flag, duration/volume, diesel/electricity + labour cost, "skipped — no water" stress flag) | [`IrrigationLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/IrrigationLogScreen.js) | `addIrrigationLog(cycleId, entry)` → `irrigationLogs[]` |
| **Scouting / pest-disease observation** (problem type, pest/disease pick-list, severity vs **ETL — Economic Threshold Level**, % affected, **photo for AI ID**, location) | [`ScoutLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/ScoutLogScreen.js) | `addObservedEvent(cycleId, entry)` → `observedEvents[]` |
| **Plant-protection spray** (control type, target, product/AI, dose ml·g per acre/pump + water vol, sprayer, PHI noted, tank-mix, product cost + bill) | **inline `SPRAY` modal** (`setModal('pesticide')`) | `addPesticide(cycleId, entry)` → `pesticidesUsed[]` |
| **Weeding / interculture** (manual weeding, hoeing, thinning, gap-fill, earthing-up, mulching — tool, area, labour + machine cost) | [`WeedingLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/WeedingLogScreen.js) | `addActivity(cycleId,{type:'WEEDING',…})` + `addLaborLog`/`addExpenseLog` |
| **Pruning / training / staking / detopping** | [`PruningLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/PruningLogScreen.js) | `addActivity(cycleId,{type:'PRUNING',…})` + cost logs |
| **Growth-stage observation & advance** (good stand/poor germination/yellowing/lodging/flower drop/water stress; vigour; photo) | `CropCycleDetail` stage control | `advanceStage(cycleId, stage)` updates `cycle.growthStage`; observations → `addObservedEvent` |
| **Ad-hoc cash out / in** (any cost or revenue not tied to the above; labour-only entries) | [`ExpenseLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/ExpenseLogScreen.js) / [`IncomeLogScreen.js`](../../../frontend/src/screens/FarmProfile/logging/IncomeLogScreen.js) | `addExpenseLog` / `addIncomeLog`; labour → `addLaborLog` |

> **Fertiliser — the N-P-K split (agronomy).** Don't treat every dose the same: **P and K are normally basal-only** (full dose at/just before sowing as DAP/SSP + MOP), while **N is split** — part basal, the rest top-dressed as urea in 1–2 splits **tied to stage**. **Zn (ZnSO₄) basal is routine in India.** *Worked example (wheat):* basal **DAP + MOP + ZnSO₄ at sowing**, then **urea top-dress at CRI (~21 DAS)** and again **at jointing**. So the modal's Basal vs Top-dress flag is agronomically meaningful, not cosmetic.

> **Irrigation — schedule around critical stages (agronomy).** Water at the moisture-sensitive phases, not on a fixed calendar: for **wheat** the critical irrigations are **CRI (~21 DAS)**, then tillering, jointing, flowering and milk/grain-fill; **rice** needs standing water from tillering through flowering. The "criticality flag" on the irrigation log is for exactly this. (Pre-sowing **palewa/rauni** is logged under Stage 2 — Land prep.)

#### Stage 8 — HARVESTED *(harvest + post-harvest/sale + close-out)*
`HARVESTED` is the **season close-out**, not the first pick. For **multi-pick crops (cotton, chilli, vegetables)** the cycle keeps accruing harvest and sale entries — the farmer advances to `HARVESTED` only at the **final pick + sale**, when the season is done. Don't end the cycle on harvest #1.

| Real-world step | Where it's captured | `farmApi` call → data |
|---|---|---|
| **Harvest** *(repeatable per pick)* (date+DAS, pick number, maturity signs, method, yield this harvest **q (quintal)**, cumulative + per-acre yield, **moisture %**, quality grade, damage/loss, labour + machine cost) | **inline `HARVEST` modal** (`setModal('harvest')`) | `recordHarvest(cycleId, data)` → `harvestYieldKg`/`harvestYieldQuintal` (= kg/100)/`harvestYieldPerAcreKg`, `harvestQualityGrade`, `harvestMoisturePct` |
| **Drying & storage** (drying method, final moisture, storage type/duration, storage/rent cost, storage loss) | `HARVEST` modal / `ExpenseLog` | `recordHarvest` / `addExpenseLog` |
| **Sale** *(repeatable — partial sales)* (buyer/channel APMC/trader/FPO/adatiya/contract/MSP/e-NAM/direct, mandi name, qty sold, grade, **rate ₹ per QUINTAL**, sold-at-MSP, deductions commission/hamali/weighing/transport/cess/moisture-cut, **net received**, payment mode, **sale patti photo**) | **inline `SALE` modal** (`setModal('sale')`) | `recordSale(cycleId, data)` → `saleSoldQuantityKg`, `salePricePerKgInr`, `saleTotalRevenueInr`, `saleBuyerType`/`saleBuyerName`, `saleDate`, `saleMandiName` |
| **Season close-out** (total cost of cultivation, cost/acre, total revenue, **net** P/L, yield/acre, benefit–cost ratio; "lessons for next season") | auto on `CropCycleDetail` | `completeCycle(cycleId)` sets stage `HARVESTED`; `getCycleFinancials(cycleId)` surfaces `computeFinancials()` |

> **Units & net revenue (agronomy/finance).** In India **mandi rates and MSP are quoted ₹ per quintal** and yield in **quintal/acre** — so the sale price the farmer knows is ₹/quintal. The schema field is `salePricePerKgInr`, so the screen must **convert ₹/quintal → ₹/kg (÷100) before sending**; defaulting capture to "₹/kg" would invite a **100× error**. Also, **net received ≠ gross**: commission, hamali, weighing, cess and moisture-cut typically shave **~6–10%**, so P&L revenue should use the **net** (the backend's `computeFinancials` nets revenue against logged inputs, but the *sale* itself should record what the farmer actually got). *(₹/quintal entry, an explicit net-deductions block, and a sale-patti photo are not modelled yet — Notes/Future work.)*

## Services, APIs & data
- **Frontend service:** all writes go through [`farmApi.js`](../../../frontend/src/services/farmApi.js). The process touches, in order: `createFarm`/`updateFarm` (Stage 0) → `createCropCycle(farmId, data)` (Crop Plan / `PLANNING`) → `addActivity({type:'LAND_PREP'…})` (land prep) → `addActivity({type:'SOWING'…})` + `advanceStage('SOWING')` + **`updateCropCycle(cycleId,{sowingDate})`** (sowing — this is what starts DAS; `updateCropCycle` → `PATCH /cycles/:id`) → per-op logs `addLaborLog`, `addExpenseLog`, `addIncomeLog`, `addObservedEvent`, `addFertilizer`, `addPesticide`, `addIrrigationLog`, `advanceStage(cycleId, stage)` → `recordHarvest` / `recordSale` / `completeCycle` / `getCycleFinancials` (Stage 8).
- **Data model:** the entire process accretes onto **one Prisma row** — `FarmCropCycle` in [`schema.prisma`](../../../backend/prisma/schema.prisma): header (`season`, `year`, `cropName`, `variety`, `isHybrid`, `isOrganic`, `areaAllocatedAcres`, `notes`); the `growthStage` enum; `sowingDate`/`actualHarvestDate`; the seed block (`seedName`, `seedBrand`, `seedSource`, `seedQuantityKg`, `seedTotalCostInr`, `seedTreatment`, `seedTreatmentProduct`); the repeatable JSON arrays (`fertilizersUsed[]`, `pesticidesUsed[]`, `irrigationLogs[]`, `observedEvents[]`, `activities[]`, `laborLogs[]`, `expenseLogs[]`, `incomeLogs[]`); harvest fields (`harvestYieldKg`/`harvestYieldQuintal`/`harvestYieldPerAcreKg`, `harvestQualityGrade`, `harvestMoisturePct`); sale fields (`saleSoldQuantityKg`, `salePricePerKgInr`, `saleTotalRevenueInr`, `saleBuyerType`/`saleBuyerName`, `saleDate`, `saleMandiName`). **Schema columns that exist but no screen writes (gaps):** `cropNameMr`/`cropNameHi`, `cropCategory`, `expectedHarvestDate`, `seedCostPerKgInr`, `seedPurchaseDate`, `seedReceiptUrl`. The backend `createCropCycle` *will* persist them if sent (and already persists `seedTreatment`/`seedTreatmentProduct`/`seedPurchaseDate`/`notes`).
- **Financials / close-out:** `computeFinancials(cycle)` in [`cropCycle.service.js`](../../../backend/src/services/cropCycle.service.js) derives cost of cultivation, cost/acre, revenue, net P/L, yield/acre and benefit–cost ratio from the accumulated cost logs; surfaced via `getCycleFinancials()`. `completeCycle()` sets the final `HARVESTED` stage.
- **Stage spine & theme:** stage order from [`StageTimelineBar.js`](../../../frontend/src/screens/FarmProfile/ui/StageTimelineBar.js); the 12 activity types + colours from `ACTIVITY_TYPES` in [`cosmicTheme.js`](../../../frontend/src/screens/FarmProfile/theme/cosmicTheme.js) (re-mapped to the KhetAI palette — reference [`khetTheme.js`](../../../frontend/src/constants/khetTheme.js) and [`LoginScreen.js`](../../../frontend/src/screens/Auth/LoginScreen.js)).

## Languages / i18n
All My Farm process strings live under the **`myFarm.v2.*`** namespace (e.g. `myFarm.v2.activity.landPrep` … `myFarm.v2.activity.income`, per `ACTIVITY_TYPES` in [`cosmicTheme.js`](../../../frontend/src/screens/FarmProfile/theme/cosmicTheme.js)), keyed via `useLanguage().t()`; the crop picker fuzzy-matches local-language synonyms (bhendi/kapas/soya). The record has **`cropNameMr` / `cropNameHi`** columns for trilingual display, but **no screen writes them today** — only `cropName` is stored — so the timeline/dashboard/Growth Story fall back to that single name. **Voice-to-text on note fields, the bill/receipt photo and the on-credit (udhaar) toggle are the intended India-first direction, not yet wired** across the loggers — treat as Notes/Future work.

## Notes, edge cases & gaps
- **This is a reference, not a screen.** There is no `CropCycleProcess` route; the process is realised across screens 04 → 05 → 06/07/08, replayed by 09. Treat this doc as the index that ties the per-stage capture points together.
- **Authoritative vs. derived stage.** The live stage is **always** `cycle.growthStage` (moved by `advanceStage`). DAS→stage bands are approximate and used **only** for Growth Story bucketing / hero scene — never as agronomic truth, and they are crop- and sowing-date-sensitive (late sowing compresses them). The whole VEGETATIVE→MATURITY band has **no stage-locked screens**; ops are logged at any DAS and the farmer advances the stage manually.
- **The Crop Plan ships, and now captures prev-crop / field-prep / water / seed-treatment.** The shipped [`CropCycleCreateScreen.js`](../../../frontend/src/screens/FarmProfile/CropCycleCreateScreen.js) is a **single-scroll** guided plan (eight sections; required = crop+season+area). It sends `cropName`, `variety` (also as `seedName`), `season`, `year`, `areaAllocatedAcres`, `isHybrid`, `isOrganic`, `seedBrand`, `seedSource`, `seedQuantityKg`, `seedTotalCostInr`, `seedTreatment`, `seedTreatmentProduct`, and a composed `notes` line (previous crop · field prep · water). It creates the cycle in `PLANNING`, **does not set `sowingDate`** (DAS starts when sowing is logged), and on success `navigation.replace('CropCycleDetail', { cycleId })`. See doc [05](05-CropCyclePlanPage.md).
- **`sowingDate` is stamped at sowing.** [`SowingLogScreen`](../../../frontend/src/screens/FarmProfile/logging/SowingLogScreen.js) calls `advanceStage('SOWING')` then `updateCropCycle(cycleId,{sowingDate:now})` — so DAS is null between planning and the first sowing log.
- **Five ops are inline modals from the quick-log rail.** `IRRIGATION`, `FERTILIZER`, `SPRAY`, `HARVEST`, `SALE` are bottom sheets on [`CropCycleDetailScreen.js`](../../../frontend/src/screens/FarmProfile/CropCycleDetailScreen.js) (`setModal(…)`); the screen auto-opens one from `route.params.prefillActivity` (the picker grid passes only `FERTILIZER`/`SPRAY`/`HARVEST`/`SALE`, routing irrigation to its own `logging/` screen). The other rail ops (Land prep, Sow, Weed, Prune, Scout, Expense, Income) push a dedicated `logging/*` screen.
- **Crop Plan = plan; loggers = dated events.** The Crop Plan records the **field-prep plan** and **seed block** once; the dated land-prep and sowing **events** (and any repeats — extra tillage, re-sowing, gap-filling) come from `LandPrepLog` / `SowingLog`.
- **`HARVESTED` is close-out, not first pick.** Both `recordHarvest` and `recordSale` are repeatable — multi-pick crops (cotton, chilli, vegetables) and staggered mandi sales keep accruing yield/net revenue; advance to `HARVESTED` only at the final pick + sale.
- **Units & money.** Area is **acres-only** today (guntha/bigha/kanal/hectare conversion is future work — and **bigha is region-variable**, so any conversion must be state-aware). Mandi/MSP prices are **₹/quintal**; the schema stores `salePricePerKgInr`, so capture must convert ₹/quintal→₹/kg (÷100) to avoid a 100× error. P&L revenue should be **net** of mandi deductions (~6–10%).
- **Server-side area guard.** `createCropCycle` can fail if `areaAllocatedAcres` exceeds the farm's remaining size; the screen surfaces the backend's exact message.
- **Schema-only gaps to keep flagging:** `seedReceiptUrl` (bill photo), `expectedHarvestDate`, `cropCategory`, `cropNameMr`/`cropNameHi`, `seedCostPerKgInr`, `seedPurchaseDate` — columns exist but no screen writes them. Also future work: offline-first sync (`SyncBadge`), per-cost bill-photo/GPS/voice-note/udhaar affordances, the structured seed-treatment order + dose, ₹/quintal entry, and per-screen Login layout chrome (Fraunces serif headers + 56px gradient icon-square) on screens that don't have it yet.
