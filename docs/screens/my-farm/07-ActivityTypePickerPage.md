# Activity Type Picker

> **Tab:** My Farm · **Stack:** `MyFarmStack` (also re-registered in `AIStack` and `ProfileStack` for deep-linking) · **Route name:** `ActivityTypePicker` · **File:** [ActivityTypePickerScreen.js](../../../frontend/src/screens/FarmProfile/ActivityTypePickerScreen.js)

## Purpose
The router screen a farmer lands on when they tap **"Log today's activity"** — a 3-column grid of **12 activity tiles** (one per `ACTIVITY_TYPE`) that maps each field operation to the right capture surface. It is deliberately thin: it holds no form fields of its own. Its only job is to take the farmer from "I just did something on the field" to the correct **dedicated logger screen** (8 of them) or the correct **inline bottom-sheet modal** on [Crop Cycle Detail](06-CropCycleDetailPage.md) (4 of them), carrying the `farmId` / `cycleId` / `plotId` scope through so the resulting log attaches to the right crop cycle. In the re-themed "My Farm" model this is the **hub for the ongoing, repeatable, event-based log** — every dated operation after sowing (each irrigation, each fertilizer dose, each spray, each weeding pass) starts here, in contrast to the one-time pre-seeding guided [Crop Plan](05-CropCyclePlanPage.md) (`CropCycleCreate`) that creates the cycle. The 12 tiles cover the full season: from `LAND_PREP` and `SOWING` at the top, through the in-season operations, down to the cross-cutting cash logs (`EXPENSE` / `INCOME`).

## Where it sits / how you reach it
- **Reached from:**
  - [MyFarmHomeScreen.js](../../../frontend/src/screens/FarmProfile/MyFarmHomeScreen.js) — the **"Log today"** quick-log rail. The "More" (`grid-outline`) chip calls `goActivityPicker(null)` → `navigate('ActivityTypePicker', { farmId: activeFarmId, cycleId: activeCycleId })`. Note the rail short-circuits: tapping the **Irrigation** chip jumps straight to `ActivityIrrigationLog` (the most-logged action) and never opens the picker; the other rail chips and "More" funnel through here. With no active farm, the rail diverts to `FarmAddEdit` instead.
  - [CropCycleDetailScreen.js](../../../frontend/src/screens/FarmProfile/CropCycleDetailScreen.js) — the **"Log activity"** section (the `SectionLabel title="Log activity"`), the primary entry when working inside a specific cycle (passes that cycle's `cycleId`).
  - Deep-links: the route is registered in `MyFarmStack`, `AIStack`, and `ProfileStack` in [AppNavigator.js](../../../frontend/src/navigation/AppNavigator.js) (lines 428 / 372 / 479), so "log an activity" round-trips can land here from the AI tab and the Profile stack.
- **Navigates to** (the routing table in [ActivityTypePickerScreen.js](../../../frontend/src/screens/FarmProfile/ActivityTypePickerScreen.js), `LOGGER_ROUTE` + the `pick(type)` handler):
  - **8 dedicated loggers** (pushed screens), each passed `{ farmId, cycleId, plotId }`:
    - `LAND_PREP` → `ActivityLandPrepLog` ([LandPrepLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/LandPrepLogScreen.js))
    - `SOWING` → `ActivitySowingLog` ([SowingLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/SowingLogScreen.js))
    - `IRRIGATION` → `ActivityIrrigationLog` ([IrrigationLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/IrrigationLogScreen.js))
    - `SCOUT` → `ActivityScoutLog` ([ScoutLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/ScoutLogScreen.js))
    - `WEEDING` → `ActivityWeedingLog` ([WeedingLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/WeedingLogScreen.js))
    - `PRUNING` → `ActivityPruningLog` ([PruningLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/PruningLogScreen.js))
    - `EXPENSE` → `ActivityExpenseLog` ([ExpenseLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/ExpenseLogScreen.js))
    - `INCOME` → `ActivityIncomeLog` ([IncomeLogScreen.js](../../../frontend/src/screens/FarmProfile/logging/IncomeLogScreen.js))
    - (See the single sectioned [Activity Loggers doc](08-ActivityLoggers.md) for these 8 screens.)
  - **4 inline modals** (not routes — reskinned bottom sheets on the cycle screen): for `FERTILIZER`, `SPRAY`, `HARVEST`, `SALE`, **when a `cycleId` is present**, it navigates `CropCycleDetail` with `{ cycleId, prefillActivity: type }`. `CropCycleDetailScreen` **consumes** `prefillActivity`: its `PREFILL_MODAL` map (`FERTILIZER→fertilizer`, `SPRAY→pesticide`, `HARVEST→harvest`, `SALE→sale`) seeds the modal state on arrival (`useState(PREFILL_MODAL[prefillActivity] || null)`), so the matching bottom sheet **auto-opens** the moment the cycle screen mounts — no extra tap. See [Crop Cycle Detail](06-CropCycleDetailPage.md) for the 4 modals.
  - **No-cycle fallback:** if one of those four is picked with **no `cycleId`** in scope, the screen shows an `Alert` — _"Open a crop cycle first, then log this activity from the cycle screen."_ — and goes nowhere (these four operations only make sense against a specific cycle).
- **Route params in:** all optional — `farmId` (scope logs to this farm), `cycleId` (scope to a specific crop cycle; sets `cycleId` on the resulting log and is what unlocks the 4 inline modals), `plotId` (v2 plot scope, threaded through to the loggers). All three are forwarded verbatim to the chosen logger.

## How it works
- The screen reads `{ farmId, cycleId, plotId }` from `route.params`, pulls the `activeFarm` from [MultiFarmContext](../../../frontend/src/context/MultiFarmContext.js), and renders a **context subtitle** in the header — `"<farm name> · this cycle"` / `"· this farm"` / `"· all farms"` depending on what scope was passed — so the farmer can confirm where the log will land before tapping.
- The body is a friendly intro bubble ("Pick what you did today / Tap any activity — fields, photos and cost will be captured for that log.") above a `flexWrap` grid that maps `ACTIVITY_TYPES` from [cosmicTheme.js](../../../frontend/src/screens/FarmProfile/theme/cosmicTheme.js) to `ActivityChip.Tile` cells (3 per row, `flexBasis: '31.5%'`, `minWidth: 104`). Each tile renders that type's **semantic accent colour** + Ionicon (icon bubble + 2-line wrapped label) from `activityMeta(key)` — so the colour vocabulary stays identical to the activity feed rows and the quick-log rail.
- **The single decision** lives in `pick(type)`: (1) if `LOGGER_ROUTE[type]` exists → `navigation.navigate(route, { farmId, cycleId, plotId })`; (2) else if `type ∈ {FERTILIZER, SPRAY, HARVEST, SALE}` **and** `cycleId` is set → `navigate('CropCycleDetail', { cycleId, prefillActivity: type })`; (3) else → the "open a cycle first" `Alert`. Tile taps fire a light haptic via `ActivityChip.Tile`.
- Tile **labels** resolve through `t(a.i18n, TYPE_LABELS[a.key] || a.key)` — i.e. the `myFarm.v2.activity.*` key with an English `TYPE_LABELS` fallback (Land prep, Sowing, Irrigation, Fertilizer, Spray, Scouting, Weeding, Pruning, Harvest, Sale, Expense, Income), so the grid renders even before the v2 strings ship.
- **(Redesigned — KhetAI re-theme)** The screen already imports the remapped [cosmicTheme.js](../../../frontend/src/screens/FarmProfile/theme/cosmicTheme.js) tokens (so the canvas, border, and text already read in the KhetAI forest-green/Plus-Jakarta family), but the header and intro are still the legacy `CosmicHeader` + plain bubble. The intended look is the Login design system: `CosmicBackground` in `GRADIENT.surface` (`#f8fef4→#e1f6dc`), a 56px `gradPrimary` icon-square in the header, and a big Fraunces serif title with an italic forest-green second line (e.g. _"What did you / *do today?*"_). Gold `#e0af3b` stays reserved for the `HARVEST` / `SCOUT` tiles and highlights only — never a primary CTA here. The 12 tiles keep their per-type accent colours; only the header/intro chrome and fonts are re-skinned.

## Services, APIs & data
- **No API calls of its own.** This is a pure navigation/router screen — it neither reads nor writes the backend. All the actual capture and persistence happens on the screen it routes to. The downstream writes (for context) go through [farmApi.js](../../../frontend/src/services/farmApi.js): the loggers/modals call `addActivity`, `addLaborLog`, `addExpenseLog`, `addIncomeLog`, `addObservedEvent`, `addIrrigationLog`, `addFertilizer`, `addPesticide`, `recordHarvest`, or `recordSale`, appending to the JSON event arrays on the `FarmCropCycle` row (`activities[]`, `irrigationLogs[]`, `fertilizersUsed[]`, `pesticidesUsed[]`, `observedEvents[]`, `laborLogs[]`, `expenseLogs[]`, `incomeLogs[]`) in [schema.prisma](../../../backend/prisma/schema.prisma).
- **Context / state:** `useLanguage()` (`t`) and `useMultiFarm()` (`activeFarm`, for the subtitle). No local mutable state.
- **Static data:** `ACTIVITY_TYPES` (the source of truth for tile order, colour, icon, and i18n key), `activityMeta`, and theme tokens (`COSMIC`, `CR`, `CS`, `CT`) from [cosmicTheme.js](../../../frontend/src/screens/FarmProfile/theme/cosmicTheme.js); the local `LOGGER_ROUTE` map (type → route name) and `TYPE_LABELS` fallback table; `Haptics` (via `ActivityChip.Tile`).
- **Navigation contract:** the 8 logger route names must stay registered in every stack the picker lives in; the 4 inline types depend on `CropCycleDetail` honouring the `prefillActivity` param — which it now does (its `PREFILL_MODAL` map auto-opens the matching modal on mount, so the handoff is end-to-end live, not a dangling param).

## Languages / i18n
- The title uses `t('myFarm.v2.pickActivity', 'What did you do?')` and each tile uses `t(a.i18n, …)` against the **`myFarm.v2.activity.*`** namespace (`landPrep`, `sowing`, `irrigation`, `fertilizer`, `spray`, `scout`, `weeding`, `pruning`, `harvest`, `sale`, `expense`, `income`). Every call carries an English fallback (`TYPE_LABELS`), so missing translations degrade to English rather than showing raw keys.
- **Hardcoded English today:** the intro bubble copy ("Pick what you did today", the helper line), the context subtitle fragments ("this cycle" / "this farm" / "all farms"), and the no-cycle `Alert` body are literal English strings, not `t()` keys — these need to move under `myFarm.v2.*` in the re-theme.
- **(Redesigned)** Adds the in-header glass language-toggle pill consistent with the rest of My Farm; tile labels can render in the farmer's chosen language once `myFarm.v2.activity.*` is populated for Hindi/Marathi.

## Notes, edge cases & gaps
- **Two distinct shapes meet here.** This grid is the entry to the **ongoing, repeatable event log** (each operation recurs across the season). It is _not_ how a cycle is created — that is the one-time pre-seeding guided [Crop Plan](05-CropCyclePlanPage.md) (`CropCycleCreate`, a single ScrollView, not a multi-step wizard), which captures the planting setup (crop, season, area, variety, previous crop / field prep, seed details, seed treatment) inline. After the cycle exists, the `LAND_PREP` / `SOWING` tiles here add the dated field passes (a harrowing, the sowing event, gap-filling / re-sowing) — the Crop Plan sets the cycle up, this logger records the operations against it.
- **`cycleId` gates 4 tiles.** Fertilizer, Spray, Harvest, and Sale are inline modals that require a specific cycle; without one they only show the "open a cycle first" `Alert`. With a `cycleId` the picker navigates to `CropCycleDetail` and the right sheet auto-opens (the `prefillActivity` round-trip is fully wired — see above), so the farmer lands directly in the form. The other 8 loggers can be reached with just a `farmId` (or even none) — but a log with no `cycleId` is harder to attribute and the loggers themselves may then need to ask which cycle. Practically, the farmer should almost always arrive here from a cycle.
- **Irrigation is the fast path.** From the home rail, Irrigation bypasses the picker entirely (it is the most-frequent operation); the picker is the comprehensive fallback. Document that the same `IRRIGATION` tile here still routes to the same `ActivityIrrigationLog`.
- **All 12 tiles always render** even if a logger isn't fully built — the original "Coming soon" sheet for unimplemented types is no longer needed now the 8 loggers + 4 modals exist, but the grid is intentionally kept visually complete so the vocabulary is stable. Any newly-added activity type just needs an `ACTIVITY_TYPES` entry plus either a `LOGGER_ROUTE` mapping or membership in the inline-modal set.
- **Stage is not enforced here.** The picker lets a farmer log _any_ activity at _any_ time regardless of `cycle.growthStage` or DAS (e.g. a spray during VEGETATIVE, a harvest pick during MATURITY). The authoritative stage stays whatever `advanceStage` last set; the picker never derives or changes it. Stage is used only for display/bucketing downstream (see [Growth Story](09-GrowthStoryPage.md)).
- **Redesign (KhetAI re-theme) — partially in code.** The screen already consumes the remapped `cosmicTheme` tokens, but the header chrome (legacy `CosmicHeader`, plain intro bubble) is not yet the full Login kit — the 56px `gradPrimary` icon-square header, Fraunces serif title with italic second line, and `CosmicBackground` surface gradient described above are intended but not yet present. The per-tile accent colours and grid layout are unchanged by the re-theme.
- **DAS / cost capture invariants live downstream.** This screen does not collect any of the global capture set (date, area, labour, cost, photo, GPS, voice-note, on-credit/udhaar toggle, bill prompt) — every one of those is captured on the logger/modal it routes to. Mentioned here only so writers don't expect cost fields on the picker itself.
- **India-specific framing.** Tile vocabulary maps to real Indian field operations (land prep / interculture-weeding / scouting / spray with PHI awareness / mandi sale). The cash tiles (`EXPENSE` / `INCOME`) exist so a farmer can log any rupee in/out — e.g. diesel, labour wages, an adatiya commission — that isn't tied to one of the structured operations, feeding the season cost-of-cultivation and net P/L computed by `computeFinancials()` in [cropCycle.service.js](../../../backend/src/services/cropCycle.service.js).
