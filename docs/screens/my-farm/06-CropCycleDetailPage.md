# Crop Cycle Detail (Season Dashboard)

> **Tab:** My Farm · **Stack:** `MyFarmStack` (also re-registered in `AIStack` for deep-linking) · **Route name:** `CropCycleDetail` · **File:** [CropCycleDetailScreen.js](../../../frontend/src/screens/FarmProfile/CropCycleDetailScreen.js)

## Purpose
The single-cycle command centre: one crop, one plot, one season. It is where the farmer lives **after** the [Crop Plan](05-CropCyclePlanPage.md) puts the crop in the ground — the place to see how far the crop has come (growth ring + 8-stage timeline), what it has cost and earned so far (Profit & Loss donut, per-acre, ROI), today's mandi rate for the crop, and the full running diary of every operation. It is also the launch pad for **logging** every farm operation (the full 12-chip quick-log rail — seven dedicated loggers + five inline bottom sheets), for the **Plan & seed** recap of the pre-seeding capture, for opening the **Growth story**, and for closing the book on the season (complete / delete).

A crop cycle moves through **8 growth stages in fixed order** — the canonical order lives in [StageTimelineBar.js](../../../frontend/src/screens/FarmProfile/ui/StageTimelineBar.js):

```
PLANNING → LAND_PREP → SOWING → VEGETATIVE → FLOWERING → FRUITING → MATURITY → HARVESTED
```

The live stage shown here is always the **authoritative** `cycle.growthStage` (set manually via `advanceStage`), never a value derived from the calendar. **DAS** (Days After Sowing) `= sowingDate ? max(0, floor((now − sowingDate)/86400000)) : null` is the timeline spine — it drives the hero stat, the stage bar caption, and (downstream) the [Growth Story](09-GrowthStoryPage.md). Before sowing, DAS is `null` and renders as `—`. (`sowingDate` is now stamped by the Sowing logger right after it advances the stage, so DAS starts counting the moment the crop is sown — see [08-ActivityLoggers.md](08-ActivityLoggers.md).)

> **Stage labels are horticulture-leaning.** The enum's `FRUITING` reads naturally for vegetables and fruit, but for other crops it should be read per category — **grain filling** (cereals), **pod development** (pulses/oilseeds), **boll development** (cotton), **fruit set/development** (veg/fruit). The screen shows the raw enum today; a per-category label map is *future work*. Likewise `HARVESTED` means **season close-out** (final pick + final sale), not the first pick — for multi-pick crops (cotton, chilli, most vegetables) the farmer keeps logging Harvest/Sale to accumulate yield **without** ending the cycle, and only marks complete at the last pick.

## Where it sits / how you reach it
- **Reached from:**
  - [MyFarmHomeScreen.js](../../../frontend/src/screens/FarmProfile/MyFarmHomeScreen.js) — the active-season hero / cycle card (`navigation.navigate('CropCycleDetail', { cycleId })`).
  - [FarmDetailScreen.js](../../../frontend/src/screens/FarmProfile/FarmDetailScreen.js) — tapping any crop-cycle row for that farm.
  - [CropCycleCreateScreen.js](../../../frontend/src/screens/FarmProfile/CropCycleCreateScreen.js) — on submitting the guided Crop Plan, it does `navigation.replace('CropCycleDetail', { cycleId })`, landing the farmer straight in the new cycle's detail.
  - The activity loggers under [logging/](08-ActivityLoggers.md) and the type picker — several hand back a `prefillActivity` param so the matching inline modal opens on arrival (see below).
  - `AIStack` deep-links — the route is re-registered there so KhetAI "open this cycle" cards can land here.
- **Navigates to:**
  - The seven pushed activity loggers under [logging/](08-ActivityLoggers.md) — **Land prep, Sow, Weed, Prune, Scout, Expense, Income** — each opened directly from its quick-log chip with `{ cycleId, farmId }`.
  - [GrowthStory](09-GrowthStoryPage.md) — the **"Growth story"** card, `navigation.navigate('GrowthStory', { cycleId, cycle })`. The `GrowthStory` route is registered in both `MyFarmStack` and `AIStack`. **This is now a real, reachable button — no longer a planned/missing CTA.**
  - Back (`navigation.goBack()`) — the header back arrow, and on a successful **delete** (the parent list/home re-loads on focus).
- **Route params in:** `{ cycleId, prefillActivity? }`. `cycleId` is the spine — everything else is fetched from it, so the screen shows a full-screen spinner until `getCropCycle` resolves. `prefillActivity` is **consumed**: on mount the screen maps it to the matching inline modal (`FERTILIZER → fertilizer`, `SPRAY → pesticide`, `HARVEST → harvest`, `SALE → sale`) and opens that sheet immediately — so a picker/logger that hands off one of those four types lands the farmer on an already-open form. **This is wired now** (`modal` initialises to `PREFILL_MODAL[prefillActivity] || null`); it is no longer a broken handoff.

## How it works
- **Load (`load`) + reload-on-focus.** `load` fires two requests in parallel: `farmApi.getCropCycle(cycleId)` (the cycle, with all its JSON log arrays) and `farmApi.getCycleFinancials(cycleId)` (computed P&L; failure is swallowed → `null`). A failure on the cycle itself shows an `Alert`. It runs **on every screen focus** (`useFocusEffect(useCallback(() => { load(); }, [load]))`) and on pull-to-refresh — so an activity logged in a dedicated pushed logger (then dismissed) appears the moment the farmer returns, not only after a manual pull. While loading it renders a `CosmicHeader` "Loading…" + centred spinner; a missing cycle renders a "Not found" state with a "Go back" `GlowButton`.
- **Derived values.** `stage = cycle.growthStage || 'PLANNING'`; `area = areaAllocatedAcres.toFixed(2)`; `das` per the formula above. Financials prefer the computed `financials.*` and fall back to fields persisted on the cycle: `totalCost = financials.totalInputCostInr ?? cycle.totalInputCostInr`, `revenue = financials.revenue ?? cycle.grossIncomeInr`, `net = financials.netProfitInr ?? (revenue − totalCost)`. `showPL` is true only when cost **or** revenue > 0, so the P&L block is hidden for a brand-new cycle. `costSegments` = `financials.costBreakdown` filtered to positive values (feeds the donut). `isCompleted = cycle.status === 'COMPLETED'`.
- **Activity feed (`buildActivityFeed`, memoised).** A pure merge of every log source on the cycle into one chronological list, newest-first. It pulls from `irrigationLogs[]`, `fertilizersUsed[]`, `pesticidesUsed[]` (→ `SPRAY`), `observedEvents[]` (→ `SCOUT`), the generic `activities[]` (land-prep / sowing / scout / weeding / pruning, typed by `it.type`), `laborLogs[]` + `expenseLogs[]` (→ `EXPENSE`), `incomeLogs[]` (→ `INCOME`), plus synthesised **Harvest** and **Sale** rows derived from the cycle's harvest/sale columns. Each row gets a title, a `·`-joined subtitle (qty, method, cost in compact ₹), and a timestamp resolved by `pickDate` (tries `date / occurredAt / timestamp / createdAt / applicationDate / loggedAt / dateTime / appliedOn`, then `cycle.updatedAt/createdAt`, then `now`) so **no entry ever silently disappears**. Rows render through `TimelineRow`, each with the activity's accent colour + Ionicon from `activityMeta(type)`.
- **Quick-log rail (full 12, hard-coded in crop-cycle order).** A horizontal scroller of `ActivityChip`s, laid out in real season order so the whole crop calendar is one tap away: **Land prep · Sow · Water · Fertilize · Spray · Weed · Prune · Scout · Harvest · Sale · Expense · Income**. Each chip's accent/icon comes from `activityMeta(type)`. The split is:
  - **Seven push a dedicated logger** (`navigation.navigate(...)` with `{ cycleId, farmId }`): Land prep → `ActivityLandPrepLog`, Sow → `ActivitySowingLog`, Weed → `ActivityWeedingLog`, Prune → `ActivityPruningLog`, Scout → `ActivityScoutLog`, Expense → `ActivityExpenseLog`, Income → `ActivityIncomeLog`.
  - **Five open an inline bottom sheet** on this screen (set `formData={}` then `modal`): Water → `irrigation`, Fertilize → `fertilizer`, Spray → `pesticide`, Harvest → `harvest`, Sale → `sale`.
  The rail is hidden entirely once the cycle is completed. (The screen opens these chips directly; there is no intermediate `ActivityTypePicker` step in this screen's flow.)
- **Inline modals (`InputModal` + `submitModal`).** Five operations are **reskinned bottom sheets, not pushed screens** — `irrigation`, `fertilizer`, `pesticide` (Spray), `harvest`, `sale`. Each is a slide-up `Modal` with a tinted icon badge, a handful of `TextInput` fields, and a "Save log" `GlowButton`. `submitModal` switches on the open modal and calls the matching `farmApi` writer (`addIrrigationLog` / `addFertilizer` / `addPesticide` / `recordHarvest` / `recordSale`), fires success/error haptics, clears the form, closes, and re-loads. These same five are the targets of the `prefillActivity` deep-link (irrigation excepted — only `FERTILIZER/SPRAY/HARVEST/SALE` are mapped).
- **Plan & seed card.** A `GlassCard` that recaps the pre-seeding capture from the [Crop Plan](05-CropCyclePlanPage.md). It renders only when at least one source field is present, with up to three rows: (1) the composed **`cycle.notes`** line ("Previous crop: … · Field prep: … · Water: …", icon `repeat-outline`); (2) a **seed** line joining `seedBrand || seedName`, `seedSource`, and `seedQuantityKg` ("… kg", icon `pricetag-outline`); (3) a **seed-treatment** line ("Seed treated · {seedTreatmentProduct}", shown when `seedTreatment === 'treated'` or a `seedTreatmentProduct` exists, icon `shield-checkmark-outline`). It is read-only here.
- **Growth story card.** A `Pressable` row (`images-outline` badge, "Growth story" title in Fraunces, "See your crop day by day · day {DAS}" subtitle) that navigates to [GrowthStory](09-GrowthStoryPage.md) with `{ cycleId, cycle }`. Always visible (not hidden on completion).
- **Complete / delete.** Both use in-app confirmation `Modal`s (not `Alert.alert`, so they work on web too). "Mark cycle complete" → `farmApi.completeCycle(cycleId)` (sets stage `HARVESTED`, `status = COMPLETED`); "Delete crop cycle" → `farmApi.deleteCropCycle(cycleId)` then `goBack()`. Both show a spinner + disabled buttons while in flight and surface server errors as an `Alert`.

## Services, APIs & data
- **API endpoints (via [farmApi.js](../../../frontend/src/services/farmApi.js)):**
  - `getCropCycle(cycleId)` → `GET /cycles/:id` — the cycle + all JSON log arrays.
  - `getCycleFinancials(cycleId)` → `GET /cycles/:id/financials` — computed P&L (`totalInputCostInr`, `revenue`/`grossIncomeInr`, `netProfitInr`, `perAcre.{costPerAcre,revenuePerAcre,profitPerAcre}`, `roiPct`, `costBreakdown[]`).
  - Inline-modal writers (the five bottom sheets): `addFertilizer`, `addPesticide`, `addIrrigationLog`, `recordHarvest`, `recordSale` (each `POST /cycles/:id/...`).
  - `completeCycle` → `POST /cycles/:id/complete`; `deleteCropCycle` → `DELETE /cycles/:id`.
  - Pushed-logger writers (called from the dedicated [loggers](08-ActivityLoggers.md), not here directly): `addActivity` (`/activity`), `addLaborLog`, `addExpenseLog`, `addIncomeLog`, `addObservedEvent`, `advanceStage`, and `updateCropCycle(cycleId, fields)` → `PATCH /cycles/:id` (used by the Sowing logger to stamp `sowingDate`).
- **Backend route/service:** `backend/src/routes/*` for cycles; **financials** are computed by `computeFinancials()` / surfaced by `getCycleFinancials()` in [cropCycle.service.js](../../../backend/src/services/cropCycle.service.js). The data model is the `FarmCropCycle` row in [schema.prisma](../../../backend/prisma/schema.prisma) — header (`cropName`, `cropNameMr`/`cropNameHi`, `cropCategory`, `variety`, `season`, `year`, `areaAllocatedAcres`, `growthStage`, `sowingDate`, `expectedHarvestDate`), seed fields (`seedName`, `seedBrand`, `seedSource`, `seedQuantityKg`, `seedTotalCostInr`, `seedCostPerKgInr`, `seedPurchaseDate`, `seedTreatment`, `seedTreatmentProduct`, `seedReceiptUrl`), `notes`, the JSON arrays (`fertilizersUsed[]`, `pesticidesUsed[]`, `irrigationLogs[]`, `observedEvents[]`, `activities[]`, `laborLogs[]`, `expenseLogs[]`, `incomeLogs[]`), the **`photos[]` string array** (cycle-level field photos — consumed by the [Growth Story](09-GrowthStoryPage.md) photo rail via `collectPhotos(cycle)`, alongside `activities[].photoUrl` and `observedEvents[].photoUrl`), and the harvest/sale columns (`harvestYieldKg`/`harvestYieldQuintal`, `harvestQualityGrade`, `actualHarvestDate`; `saleTotalRevenueInr`, `salePricePerKgInr`, `saleBuyerName`/`saleMandiName`, `saleDate`). **Gap:** `cropNameMr`/`cropNameHi`, `cropCategory`, `expectedHarvestDate`, `seedCostPerKgInr`, `seedPurchaseDate`, and `seedReceiptUrl` are schema columns **not written by any screen** today (see Notes).
- **Mandi data:** `getMandiPrices` / `getMandiTrend` (Agmarknet) via [MandiGlanceCard.js](../../../frontend/src/screens/FarmProfile/components/MandiGlanceCard.js).
- **State / context:** `LanguageContext` (`t`), `useSafeAreaInsets`; local `useState` for `cycle`, `financials`, `modal`, `formData`, the loading/confirm flags; `useMemo` for the feed; `Haptics` util.
- **UI kit:** `CosmicScreen`, `CosmicHeader`, `GlassCard`, `GlowButton`, `StageTimelineBar`, `ActivityChip`, `SpeakerButton` (FarmProfile `ui/`); `DonutChart`, `GrowthRing` (`components/charts`); `CropIcon` (`components/CropIcons`); tokens (`COSMIC`, `CR`, `CS`, `activityMeta`) from [cosmicTheme.js](../../../frontend/src/screens/FarmProfile/theme/cosmicTheme.js).

## Languages / i18n
Most copy resolves through `useLanguage().t(...)` against the `farmProfile.*` namespace (delete/complete dialog titles + messages, load/save error fallbacks, the header delete label) with English fallbacks; generic actions reuse `rent.cancel` / `rent.delete` and `login.error`. **A large share of strings on this screen are still hardcoded English literals** — the hero stat labels ("days", "acres", "season"), every section label ("Growth stage", "Plan & seed", "Profit & loss", "Market price", "Log activity", "Recent activity"), the 12 quick-rail chip labels ("Land prep", "Sow", "Water", …), the "Growth story" card copy, the modal titles/fields ("Add fertilizer", "Product *", "Quantity (kg)", …), and the empty-feed copy. Activity-type accents/icons carry `myFarm.v2.activity.*` i18n keys in `ACTIVITY_TYPES`, but this screen does not yet read them. *Future work:* move all My Farm copy under a shared `myFarm.v2.*` namespace and add voice-to-text (local language) on note fields.

## Notes, edge cases & gaps
- **Stage is authoritative, never derived.** The hero pill, ring, and timeline all reflect the manually-set `cycle.growthStage`. DAS→stage bucketing is *only* used downstream in [Growth Story](09-GrowthStoryPage.md) to label historical events as "approximate" — never as agronomic truth on this screen. There is no inline `advanceStage` control on this screen today (stage advances happen inside the Land-prep / Sowing loggers); a "move to next stage" affordance on the growth-stage card is *future work*.
- **DAS bands are crop- and sowing-date-sensitive, not fixed.** The `StageTimelineBar` lays the 8 stages out at even visual intervals, but real DAS anchors vary by crop and shift with sowing date (late sowing compresses the calendar). For example, generic **wheat** runs roughly: tillering/VEGETATIVE ~0–60 DAS, jointing–booting ~60–80, FLOWERING (anthesis) ~80–100, grain-fill ~100–125, MATURITY ~125–140. Treat any prose example as indicative only and prefer a **per-crop DAS table** over hard-coded numbers when this gets wired into stage labelling.
- **`FRUITING` is horticulture language.** Read it per crop category — grain filling (cereals), pod development (pulses/oilseeds), boll development (cotton), fruit set/development (veg/fruit). The screen shows the raw enum; a per-category label map is *future work*.
- **Pre-sowing cycle.** A cycle saved at `PLANNING`/`LAND_PREP` (before `sowingDate`) shows DAS `—`, an empty feed, and a hidden P&L block — this is expected. The **Plan & seed** card still renders from `cycle.notes` + seed fields, and the [Growth Story](09-GrowthStoryPage.md) hero shows the planning state.
- **Inline modals vs pushed loggers.** Five operations are deliberately bottom sheets here (Water, Fertilize, Spray, Harvest, Sale); the other seven (Land prep, Sow, Weed, Prune, Scout, Expense, Income) route to a dedicated `logging/*` [logger](08-ActivityLoggers.md). The same Land-prep / Sowing fields the [Crop Plan](05-CropCyclePlanPage.md) captures as the *first* pass are added as *subsequent* passes through those loggers.
- **Fertilizer capture doesn't yet model the N-P-K split.** In Indian practice fertiliser is **split, not single-shot**: *basal* = full **P** + full **K** + part of **N** (e.g. DAP/SSP + MOP, plus routine **ZnSO₄**, at sowing); *top-dress* = the remaining **N** as urea in 1–2 splits tied to stage (wheat: urea at CRI ~21 DAS and again at jointing). P and K are normally basal-only; N is split. The inline modal records only product/qty/cost — it has no basal-vs-top-dress, stage, dose-type, method, or bill-photo field. *Future work* should let the farmer tag each fertiliser pass to a stage so the timeline reads as a real nutrient schedule. (Seed-treatment order is documented on [05-CropCyclePlanPage.md](05-CropCyclePlanPage.md): fungicide → insecticide → bio-agent/biofertiliser last and only after the chemical dressings dry; Rhizobium + PSB is near-universal for pulses/oilseeds/groundnut.)
- **Sale price unit & net-vs-gross (P&L correctness).** Mandi sale and MSP are quoted **₹ per quintal**, and yield in **quintal/acre** — the Sale modal's `pricePerKgInr` field invites a 100× error if the farmer types the per-quintal figure. The default sale unit *should* be ₹/quintal (kg optional). Equally, **net received ≠ gross**: APMC commission, *hamali*, cess and moisture-cut typically shave ~6–10% off the gross, so P&L **revenue should use the NET amount**, not the headline rate × quantity. Neither the ₹/quintal default nor a deductions/net capture exists yet. (Area is **acres-only** here; *bigha* is region-variable and would need a state-aware conversion — *future work*.)
- **`HARVESTED` = season close-out, not first pick.** For multi-pick crops (cotton, chilli, most vegetables) the farmer logs Harvest/Sale repeatedly, accumulating yield, and only marks the cycle complete at the final pick + final sale. "Mark cycle complete" sets stage `HARVESTED` + `status = COMPLETED` and hides the quick-log rail — so don't complete a multi-pick cycle until the last pick is in.
- **Scouting & thresholds.** The Scout logger ([08-ActivityLoggers.md](08-ActivityLoggers.md)) and `observedEvents[]` feed should reference **ETL (Economic Threshold Level)** — spray decisions are driven by pest counts crossing the ETL, not by a fixed calendar. The detail screen surfaces these as `SCOUT` rows but does not yet carry ETL guidance.
- **Inline modals are thin.** They capture 2–4 fields each. A richer capture catalogue (spray target/AI/dose/PHI/tank-mix; harvest pick-number/moisture %/cumulative yield; sale buyer-channel/deductions/net/receipt) plus per-cost **photo, GPS, voice-note, and "on credit (*udhaar*)?" toggle** — with **bill/receipt capture on every purchase** (needed for PMFBY claims and dealer complaints) — is *future work*. None of those extra fields exist yet.
- **Theme is already KhetAI.** The screen renders forest-green/gold + Plus Jakarta + Fraunces: `cosmicTheme` token *values* are remapped (`PRIMARY #005F21`, `ACCENT` gold `#E0AF3B`, forest-tinted text, soft-green `GLOW`/`KSHADOW`), all `Inter_*` literals were swapped to `PlusJakartaSans_*`, and the hero crop name + Growth-story title already use Fraunces serif. Remaining chrome work is per-screen Login layout (56px `gradPrimary` icon-square headers, Fraunces titles with an italic forest-green second line) — not applied to every section here yet.
- **Growth Story is shipped and reachable.** `GrowthStoryScreen.js` exists and the `GrowthStory` route is registered in both `MyFarmStack` and `AIStack`; the "Growth story" card here navigates to it with `{ cycleId, cycle }`. It has its own photo rail (`collectPhotos(cycle)` over `cycle.photos[]` + `activities[].photoUrl` + `observedEvents[].photoUrl`) and an 8-stage filmstrip — it does **not** reuse this screen's `buildActivityFeed`. Photorealistic AI stage images are *future work* (no growth-image endpoint yet).
- **Financials degrade gracefully.** `getCycleFinancials` failure is swallowed (`.catch(() => null)`); the screen then falls back to fields stored on the cycle, and the donut only renders when `costBreakdown` has positive segments. The "Spent/Earned/Profit" grid still shows from the fallbacks.
- **Feed robustness.** `pickDate` guarantees every log entry gets a usable timestamp (cascading to `now`), so malformed/undated entries still appear (sorted to "just now") rather than vanishing.
- **Unwired schema columns.** `cropNameMr`/`cropNameHi`, `cropCategory`, `expectedHarvestDate`, `seedCostPerKgInr`, `seedPurchaseDate`, and `seedReceiptUrl` exist on `FarmCropCycle` but are not written by any screen — flagged as gaps. (`seedTreatment`, `seedTreatmentProduct`, `seedPurchaseDate`, and `notes` *are* now persisted by `createCropCycle` on the backend.)
- **Offline-first** is *future work*: logs should be writable offline and synced later with a "Saved offline, will sync" notice; the current inline writers go straight to the API and surface an `Alert` on failure.
- **Delete is destructive and irreversible** — it removes the cycle and all its logs; "Complete" is non-destructive and keeps records viewable.
