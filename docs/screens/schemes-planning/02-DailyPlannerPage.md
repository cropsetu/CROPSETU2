# Daily Planner

> **Tab:** AI Assistant · **Stack:** `AINavigator` (AIStack) · **Route name:** `DailyPlanner` · **File:** `frontend/src/screens/AI/DailyPlannerScreen.js`

## Purpose
A to-do list of the day's farm tasks, grouped by priority (Urgent / Today / Planned / Done), each with crop, field, timing, action detail and an AI reasoning note. Farmers tick tasks off as they complete them and a progress bar tracks the day's completion. Tasks come from the planner backend and can be (re)generated by AI for the current crop/season.

## Where it sits / how you reach it
- **Reached from:**
  - Registered in `AppNavigator.js` as `<AIStack.Screen name="DailyPlanner" component={DailyPlannerScreen} />` (`headerShown: false`), inside the AI Assistant tab's stack.
  - No tile in `AIAssistantHome`'s `QUICK_SERVICES`/`AI_TOOLS` arrays points at `DailyPlanner`, and no `navigation.navigate('DailyPlanner')` call exists elsewhere in `frontend/src/`. It is reachable only by code that pushes the `DailyPlanner` route by name (e.g. deep link / programmatic nav); there is no wired button to it in the screens currently in the repo.
- **Navigates to:**
  - **Back** — header back chevron calls `navigation.goBack()`. There are no outbound `navigation.navigate` calls.
- **Route params in:** none (the component reads only `navigation` from props).

## How it works
- **On mount** `useEffect` → `loadTasks()` calls `getPlannerTasks()` and stores the result in `tasks`; `loading` is true during the fetch and the screen shows a spinner. On error it falls back to an empty task list.
- Key state: `tasks` (array), `loading`, `generating` (AI-generate in-flight). `today` is a localized date string (`en-IN`, weekday/day/month).
- **Progress:** `done` = tasks with a truthy `doneAt`; `total` = task count; `pct` = done/total. An `Animated.Value` (`progAnim`) animates the progress-bar fill width over 500ms whenever `pct` changes. When all tasks are done (and total > 0) an "all tasks done" row appears.
- **Grouping:** tasks are partitioned into `urgent`, `today_`, `planned` (incomplete, by `priority`) and `doneTasks` (anything with `doneAt`). `renderSection` renders a labeled section with a colored dot and count badge per non-empty group.
- **Toggle a task:** the checkbox calls `toggle(id)`, which does an **optimistic update** (sets/clears `doneAt` locally) then `await updateTaskDone(id, nowDone)`; on failure it reverts the local change.
- **Generate with AI:** the header "Regenerate" button and the empty-state "Generate with AI" button call `handleGenerate(true)`, which sets `generating`, calls `generateAITasks({ crop: 'Tomato', state: 'Maharashtra', dayOfSeason: 45 })` (context is **hard-coded**), and replaces `tasks` with the result. On failure it keeps the existing tasks.
- **Task expand:** each `TaskItem` has its own `open` state; tapping the row body animates a chevron rotation (240ms) and reveals `task.description` plus an AI-reason chip.

## Services, APIs & data
- **API endpoints (via `services/aiApi.js`):**
  - `GET /planner/tasks` — `getPlannerTasks()` (load on mount; accepts optional `date` param, not passed here).
  - `PUT /planner/tasks/:id` — `updateTaskDone(id, done)` (toggle complete).
  - `POST /planner/generate` — `generateAITasks({ crop, state, dayOfSeason })` (AI task generation, 20s timeout).
  - (`aiApi.js` also exposes `createTask` / `deleteTask` for `POST` / `DELETE /planner/tasks`, but this screen does not call them.)
- **Backend route/service:** `backend/src/routes/planner.routes.js` (handlers for `GET /tasks`, `POST /tasks`, `PUT /tasks/:id`, `DELETE /tasks/:id`, `POST /generate`). All `aiApi` requests carry the auth token via `api.js` interceptors.
- **State / context:** `useLanguage()` for `t()`; `useSafeAreaInsets()`; local `useState` (`tasks`, `loading`, `generating`) and per-item `open`/animation refs. No AuthContext / writeQueue / socket.
- **Local / static data:** an in-file `ALL_TASKS` mock array and a `PRIORITY` config map exist. `PRIORITY` is used for pill colors/labels; `ALL_TASKS` is **not rendered** — tasks come from the API (it appears to be leftover seed/sample data).

## Languages / i18n
- Uses the `planner.*` namespace. Keys seen: `title`, `generating`, `regenerate`, `tasksComplete` (`{done, total}`), `allTasksDone`, `generatingAI`, `noTasks`, `generateWithAI`, `sectionUrgent`, `sectionToday`, `sectionPlanned`, `sectionDone`, plus priority labels `urgent`, `today`, `planned`.
- The `planner` namespace is present in `src/i18n/translations.js` (English, Hindi, Marathi) and the per-language files, so UI chrome is multi-language. Task content comes from the backend.

## Notes, edge cases & gaps
- **Field-name mismatch risk:** `TaskItem` renders `task.description` for the detail and reads `task.doneAt` for completion, while the in-file mock objects use `detail` and `done` — confirming the live data shape is driven by the backend (`description`, `doneAt`), not the mock.
- **Hard-coded generation context:** AI regeneration always passes `crop: 'Tomato', state: 'Maharashtra', dayOfSeason: 45` rather than the user's real farm profile.
- **Optimistic updates:** toggling a task updates the UI immediately and reverts if the `PUT` fails; generation keeps existing tasks if it fails (both swallow errors silently — no toast/error UI).
- **No clear inbound link:** no in-app button currently navigates to this route within the repo's screens, so it may be reachable only via deep link or future wiring.
- States covered: loading spinner, empty state with generate CTA, all-done banner. No explicit network-error banner.
